# 📜 `migrations/`

## Назначение

Директория `migrations` содержит **скрипты миграций схемы базы данных**: создание таблиц, индексов, изменение полей, перенос данных и пр.

> Это **единственный источник правды** о структуре вашей базы данных.

---

## 🧱 Как работает

Миграции применяются последовательно с использованием миграционного движка  
например [`golang-migrate`](https://github.com/golang-migrate/migrate) или встроенного инструмента в `sqlc`, `ent` и др.

migrate -path ./migrations -database postgres://user:pass@localhost:5432/db up

## 🧭 Правила миграций

### 1. 🔒 Миграции неизменяемы

> После того как миграция попала в `main` — **не редактировать!**

Если нужно исправить — создайте новую миграцию с корректировкой.

### 2. 📑 Нумерация

Каждая миграция должна начинаться с порядкового номера:
- `0001_*.sql`
- `0002_*.sql`
- `...`

Миграции применяются **в лексикографическом порядке**.

### 3. ✏️ up/down

Каждая миграция должна иметь:
- `up.sql` — применяется при `migrate up`
- `down.sql` — применяется при `migrate down`

Даже если вы не планируете делать откаты — создайте `down.sql` с `DROP` / `DELETE`.

### 4. ✅ Идём строго через PR

Все миграции проходят ревью как обычный код.  
Проверяйте:
- **названия таблиц и колонок**
- **nullable/not null**
- **индексы и foreign key'и**
- **безопасность ALTER TABLE (downtime!)**

## 🧪 Тестирование миграций

В CI можно прогонять миграции "с нуля":

```bash
docker compose up -d db
migrate -path ./migrations -database $DB_URL up
go test ./integration_test/...
```
