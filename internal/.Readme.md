# 🧩 `internal/`

## Назначение

Директория `internal` содержит **всю внутреннюю бизнес-логику и архитектурные слои микросервиса**, которые **недоступны извне** (на уровне Go-пакетов).  
Это основа DDD-подхода, где реализуются ключевые сущности, use case'ы и зависимости, разделённые по смысловым модулям.

> Всё, что находится внутри `internal/`, не может быть импортировано из других микросервисов — это гарантирует модульность и изоляцию.

---

## 📦 Поддиректории

| Путь                  | Назначение                                                                 |
|------------------------|---------------------------------------------------------------------------|
| `internal/api/`        | HTTP/gRPC-интерфейсы, хендлеры, преобразование запросов/ответов.         |
| `internal/app/`        | Точка инициализации всех зависимостей и связей внутри микросервиса.       |
| `internal/domain/`     | Чистые бизнес-сущности, value-объекты и интерфейсы.                       |
| `internal/usecase/`    | Прикладная логика (Application Layer): сценарии работы, правила, шаги.   |
| `internal/repo/`       | Реализации интерфейсов доступа к хранилищам: базы данных, кэши и т.д.     |
| `internal/pkg/`        | Повторно используемые вспомогательные пакеты внутри сервиса.              |

---

## 🧱 Архитектурный подход

Следуем принципам **чистой архитектуры** (Clean Architecture) и **DDD**:
[ api ] <-- Адаптеры (входные точки)
[ usecase ] <-- Бизнес-правила, сценарии
[ domain ] <-- Сущности и контракты
[ repo ] <-- Реализации хранилищ
[ app ] <-- Сборка всего приложения

Каждый слой зависит только от более внутреннего и не знает ничего о внешних.

---

## 🧪 Тестирование

- Все публичные методы в `usecase` и `repo` должны иметь unit-тесты.
- Для хендлеров в `api` — использовать интеграционные тесты с моками зависимостей.
- Для `domain` — только unit-тесты, без внешних зависимостей.

---

## 🔐 Инкапсуляция

Модули в `internal/` **не экспортируются наружу**.  
Это означает:

- Вы не сможете сделать `import "yourservice/internal/usecase"` из других сервисов.
- Повторно используемый код, предназначенный для внешнего использования, должен находиться в `pkg/`.

---

## 🧠 Советы

- Следите за **границами ответственности** между модулями.
- Не допускайте бизнес-логику в `api/` или `repo/`.
- Упрощайте внедрение зависимостей через `app/` с минимальной связностью.
